Libro Blanco de Seguridad de WordPress

Visión de conjunto

Este documento es un análisis y explicación del desarrollo de software del núcleo de WordPress y sus procesos de seguridad relacionados, al igual que un examen de la seguridad inherente construida directamente en el software. Las personas que toman decisión y que evaluan WordPress como un sistema de gestión de contenidos o como un marco de aplicaciones web deberían usar este documento en su análisis y toma de decisión, y para los desarrolladores deberían referirse al mismo para familiarizarse con los componentes de seguridad y las mejores prácticas de software.

La información de este documento está actualizada para la última versión estable del software, WordPress 4.1 en el momento de su publicación, pero debería ser considerado relevante también para las versiones más recientes del software, ya que la compatibilidad con versiones anteriores es una prioridad en el equipo de desarrollo de WordPress. Medidas de seguridad específicas y otros cambios se anotarán a medida que se añadan al núcleo del software en versiones específicas. Se recomienda encarecidamente que siempre se ejecute la última versión estable de WordPress para asegurarse de tener la experiencia más segura posible.

Sumario Ejecutivo

WordPress es un sistema de gestión de contenidos dinámico de fuentes abiertas que se usa para hacer funcionar millones de sitios web, aplicaciones web y blogs. En la actualidad es usado por más del 23% (más de 10 millones) de los sitios web en Internet. La usabilidad, extensibilidad y la madurez de la comunidad de desarrollo hace que sea una elección popular y segura para sitios web de todos los tamaños.

Desde su concepción de 2003, WordPress ha sufrido continuos cambios para hacerlo más robusto, de manera que su núcleo pueda gestionar y mitigar las amenazas de seguridad más comunes, incluyendo la lista del Top 10 identificado por El Proyecto de Seguridad para Aplicaciones Web Abiertas (OWASP) como las vulnerabilidades de seguridad más frecuentes, que se discuten en el presente documento.

El equipo de seguridad de WordPress, en colaboración con el equipo que lidera el desarrollo del núcleo de WordPress, y respaldados por la comunidad global de WordPress, trabaja para identificar y resolver incidencias de seguridad en el núcleo de software disponible para su distribución e instalación en WordPress.org, y también para recomendar y documentar las mejores prácticas en seguridad para plugins de terceras partes y para autores de temas de terceros.

Los desarrolladores de sitios web y los administradores deberían prestar especial atención al uso correcto de las APIs del núcleo y la configuración del servidor que han sido la fuente de vulnerabilidades frecuentes, y también deben asegurarse que todos los usuarios emplean claves fuertes para acceder a WordPress.

Una visión general de WordPress

WordPress es un sistema de gestión de contenidos (CMS) de fuentes abiertas y gratuito. Es el CMS más extensamente usado en el mundo y funciona en más del 23% de los 10 millones de sitios web más vistos1, ofreciendo una cuota de mercado estimada del 60% de todos los sitios que usan un CMS.

WordPress está bajo la Licencia Publica General (GPLv2 o posterior) que proporciona 4 libertades fundamentales, y que puede considerarse como la "carta de derechos" de WordPress

  1. La libertad de ejecutar el programa, para cualquier propósito.
  2. El acceso al código fuente, libertad de estudiar cómo funciona el programa, y la libertad de cambiarlo para que haga lo que deseas.
  3. La libertad de redistribuirlo.
  4. La libertad de distribuir copias de tus versiones modificadas a otros. Si lo haces, das la oportunidad de beneficiarse de tus cambios a toda la comunidad.

El equipo líder del núcleo de WordPress

El proyecto WordPress es una meritocracia, dirigido por un equipo líder central, y conducido por su co-creador y desarrollador principal, Matt Mullenweg. El equipo gobierna todos los aspectos del proyecto, incluyendo el desarrollo del núcleo, WordPress.org, y las iniciativas de la comunidad.

El equipo que lidera el núcleo está formado por Matt Mullenweg, 5 desarrolladores principales, y más de una docena de desarrolladores del core con acceso permanente a incluir código. Estos desarrolladores tienen autoridad final en decisiones técnicas, y conducen las discusiones técnicas y los esfuerzos de implementación.

WordPress tiene un número de desarrolladores contribuyendo. Algunos de estos son antiguos o actuales commiters, y algunos son probablemente futuros commiters. Estos desarrolladores que contribuyen son contribuidores veteranos y de confianza de WordPress que se han ganado un gran respeto entre sus homólogos. Según necesidades, WordPress tiene committers invitados, que son individuos a los que se concede acceso para commit, en algunas ocasiones para componentes específicos o con criterios temporales o de prueba.

Los desarrolladores del núcleo y los contribuidores guían principalmente el desarrollo. En cada versión, cientos de desarrolladores contribuyen al código de WordPress. Estos contribuidores del core son voluntarios que contribuyen al código fuente base de alguna manera.

El ciclo de liberación de versión de WordPress

Cada ciclo de liberación de WordPress es conducido por uno o más desarrolladores del core de WordPress. Un ciclo de liberación generalmente dura unos 4 meses desde la reunión inicial hasta el lanzamiento de la versión.

Un ciclo de liberación sigue el siguiente patrón2:

  • Fase 1: Planificación y confirmación de los líderes de equipo. Esto se hace en la sala de chat #core en Slack. El desarrollador principal de la versión discute las características que se incluirán en la siguiente versión de WordPress. Los contribuidores de WordPress se implican y forman parte de la mencionada discusión. El desarrollador principal asignará líderes de equipo para cada una de las características.
  • Fase 2: Inicio del desarrollo. Los lideres de equipo montan equipos y trabajan en sus funciones o características asignadas. Se programan chats periodicos para asegurarse que el desarrollo sigue progresando.
  • Fase 3: Beta. Se liberan las Betas, y los a los beta-testers se les pide que empiecen a informar de fallos. No se realizan más commits para nuevas mejoras, y no se toman en cuenta solicitudes de funcionalidades a partir de esta fase. Se avisa a los autores de plugins y temas externos para que se aseguren de que su código funciona con la nueva versión.
  • Fase 4: Release Candidate. Se congelan las cadenas traducibles a partir de este punto (es decir, no habrá cadenas nuevas). El trabajo se centra en regresiones e incidencias críticas únicamente.
  • Fase 5: Lanzamiento. Se lanza la versión de WordPress y se hace disponible en el escritorio de WordPress para actualizaciones.

Numeración de versiones y distribuciones de seguridad

Una versión principal de WordPress se identifica por las 2 primeras secuencias. Por ejemplo, 3.5 es una versión principal, al igual que 3.6, 3.7, o 4.0. No hay un "WordPress 3" o "WordPress 4" y cada distribución o versión principal es referida por su numeración, i.e., "WordPress 3.9."

La distribuciones más importantes pueden añadir nuevas características y APIs de desarrollo. Aunque en el mundo del software, una versión "principal" significa que se pueda romper la compatibilidad con versiones precedentes, WordPress se esfuerza en no romper la compatiblidad con versiones previas. La compatibilidad hacia atrás es una de las filosofías más importante del proyecto, con el objetivo de hacer las actualizaciones más sencillas, tanto para usuarios como para desarrolladores.

Una versión menor de WordPress se indica por la tercera secuencia numérica. La versión 3.5.1 es una versión menor, al igual que la 3.4.23. Una versión menor se reserva para arreglos de vulnerabilidades de seguridad y abordar errores críticos únicamente. Ya que se liberan nuevas versiones de WordPress con frecuencia - el objetivo es que cada 4-5 meses haya una distribución mayor, y las distribuciones menores se liberan según necesidad- sólo hay necesidad de distribuciones mayores y menores.

Compatibilidad con versiones precedentes

El proyecto WordPress mantiene un fuerte compromiso con la compatibilidad con versiones precedentes. Este compromiso significa que los temas, plugins y el código personalizado continúa funcionando cuando el core de WordPress se actualiza, facilitando a los propietarios del sitios web a mantener sus WordPress actualizados a la última versión segura.

WordPress y la Seguridad

El equipo de seguridad de WordPress

El equipo de seguridad de WordPress está formado por aproximadamente 25 expertos, incluyendo a desarrolladores principales e investigadores de seguridad — aproximadamente la mitad son empleados de Automattic (los creadores de WordPress.com, la plataforma más antigua y grande de alojamiento de WordPress en la web), mientras que otros trabajan en el campo de la seguridad web. El equipo consulta con investigadores en seguridad bien conocidos y de confianza, y con empresas de alojamiento web.3.

El equipo de seguridad de WordPress a menudo colabora con otros equipos de seguridad para abordar incidencias en dependencias comunes, tales como resolver la vulnerabilidad en el intérprete de PHP XML, usado por la API de XML-RPC que se incluye con WordPress, en WordPress 3.9.24. Esta resolución de la vulnerabilidad fue el resultado de un esfuerzo conjunto entre los equipos de seguridad de WordPress y Drupal.

Riesgos de seguridad en WordPress, Proceso e Historia

El equipo de seguridad de WordPress cree en la revelación responsable de información, alertando al equipo de seguridad de manera inmediata ante cualquier vulnerabilidad potencial. Las vulnerabilidades de seguridad potenciales se pueden indicar al equipo de seguridad directamente a través de la dirección de email: security@wordpress.org5. El equipo de seguridad se comunica entre sí a través de una lista privada de correo, y trabaja en un entorno seguro y privado con Trac para seguimiento de incidencias, pruebas, y arreglos de fallos y problemas de seguridad.

Cada informe de seguridad es reconocido tras su recepción, y el equipo trabaja para verificar la vulnerabilidad y determinar su grado de severidad. Si se confirma, el equipo de seguridad planifica el parche para solventar el problema, que puede ser confirmado para la siguiente versión del software de WordPress, o que puede ser subido de inmediato como una distribución de seguridad, dependiendo de la severidad de la incidencia.

Para una distribución inmediata de seguridad, se publica un aviso por parte del equipo de seguridad en el sitio de noticias de WordPress.org6anunciando la distribución y detallando los cambios. Se da crédito al responsable del aviso para animar y reforzar a que se informe de forma responsable sobre futuras vulnerabilidades.

Los Administradores de WordPress ven una notificación en el escritorio de su sitio web para actualizar cuando hay una nueva versión disponible y, siguiendo el manual de actualización, los usuarios son redirigidos a la pantalla de Acerca de WordPress que detalla los cambios. Si los administradores tienen habilitadas las actualizaciones automáticas en segundo plano, recibirán un email después de que la actualización se haya completado.

Actualizaciones automáticas en segundo plano para versiones de seguridad

Desde la versión 3.7, WordPress introdujo actualizaciones automáticas en segundo plano para todas las versiones menores7, tales como la 3.7.1 y 3.7.2. El equipo de seguridad de WordPress puede identificar, arreglar, y promover mejoras de seguridad automáticas para WordPress sin que el propietario del sitio necesite hacer nada por su parte, y la actualización de seguridad se instalará automáticamente.

Cuando una actualización de seguridad se propone para la versión estable vigente de WordPress, el equipo principal realizará también dicha actualización de seguridad para todas las versiones que sean capaces de recibir actualizaciones en segundo plano (desde WordPress 3.7), de manera que versiones más antiguas pero recientes de WordPress reciban estas mejoras de seguridad.

Los propietarios de sitios web pueden optar por desactivar las actualizaciones automáticas en segundo plano a través de un simple cambio de su fichero de configuración, pero se recomienda mantener esta funcionalidad por parte del equipo principal, al igual que ejecutar la versión estable más reciente de WordPress.

El Top 10 de OWASP 2013

El Proyecto Abierto de Seguridad de Aplicaciones Web (OWASP) es una comunidad online dedicada a la seguridad de aplicaciones web. La lista de los 10 principales de OWASP8 se centra en la identificación de los riesgos de seguridad más importantes en aplicaciones para una amplia gama de organizaciones. Los 10 elementos más importantes son seleccionados y priorizados en combinación con la estimación consensuada sobre explotabilidad, detectabilidad e impacto.

Las siguientes secciones discuten las APIs, recursos, y políticas que usa WordPress para endurecer el núcleo de software y los plugins y temas de terceras partes contra estos riesgos potenciales.

A1 - Inyección

Hay un conjunto de funciones y APIs disponibles en WordPress para ayudar a los desarrolladores a asegurar sus sitios contra inyección de código no autorizado, y ayudarles a validar y sanitizar los datos. Las mejores prácticas y documentación sobre como usar estás APIs para proteger, validar o sanitizar la entrada y salida de datos en HTML, URLs, cabeceras HTTP, y cuando se interacciona con la base de datos o el sistema de ficheros están disponibles9. Los Administradores pueden además restringir los tipos de fichero que se pueden subir a su sitio web mediante filtros.

A2 - Rotura de autenticación y gestión de sesiones.

El software del núcleo de WordPress gestiona las cuentas de usuario y la autenticación así como los detalles tales como el identificador del usuario, nombre y contraseña, que son gestionados en el lado del servidor, al igual que las cookies de autenticación. Las contraseñas están protegidas en la base de datos usando las técnicas estándares de salting y streching. Las sesiones existentes se destruyen al salir del sistema para las versiones de WordPress posteriores a la 4.0.

A3 - Cross Site Scripting (XSS)

WordPress proporciona una rango de funciones que pueden ayudar a asegurar que los datos suministrados por el usuario son seguros10. Los usuarios de confianza, esto es, los administradores y editores de una instalación sencilla de WordPress, y los administradores de sitios Multisite, pueden publicar HTML o JavaScript sin filtrar si lo necesitan dentro de un post o una página. El contenido de los usuarios que no son de confianza, así como el contenido enviado por los usuarios, se filtra por defecto para eliminar las entidades peligrosas usando la biblioteca KSES mediante la función wp_kses.

Como ejemplo, el equipo principal de WordPress avisó antes de la versión de WordPress 2.3 que la función the_search_query() estaba siendo mal usada por la mayor parte de autores de temas, que no estaban escapando la salida de la función para su uso en HTML. En casos muy raros de pequeñas roturas en la compatibilidad con versiones previas, la salida de la función fue cambiada en WordPress 2.3 para que fuera pre-escapada.

A4 - Referencia a objectos directos inseguros

WordPress a menudo proporciona referencias directas a objetos, tales como identificadores numéricos únicos de cuentas de usuario o contenido disponible en la URL o campos de formularios. Mientras que estos identificadores revelan información directa del sistema, el sistema de control de acceso y permisos de WordPress previene las solicitudes no autorizadas.

A5 - Mala configuración de Seguridad

La mayor parte de las operaciones de configuración de seguridad están limitadas a un solo administrador autorizado. La configuración por defecto de WordPress es evaluada de manera continuada desde el equipo principal, y el equipo principal de WordPress proporciona documentación y las mejores prácticas para reforzar la seguridad de la configuración del servidor que ejecuta un sitio WordPress11.

A6 - Revelación de datos sensibles

Las contraseñas de cuentas de usuario de WordPress están basadas en salt y hash de Portable PHP Password Hashing Framework12. El sistema de permisos de WordPress se usa para control de acceso a la información privada, tales como información personal de identificación de usuarios registrados (PII), direcciones de correo de los participantes, contenido publicado de manera privada, etc. En WordPress 3.7, se incluyó un medidor de la fortaleza de la contraseña en el núcleo del software, proporcionando información adicional a los usuarios cuando se asignan sus contraseñas, y sugerencias para incrementar dicha fortaleza. WordPress también cuenta con una configuración opcional para requerir HTTPS.

A7 - Control de acceso a nivel de funciones

WordPress comprueba la adecuada autorización y permisos en cualquier función de solicitud de nivel de acceso antes de que la acción se lleve a cabo. El acceso o la visualización de URLs administrativas, menús, y páginas está ligado e integrado con el sistema de autenticación. Sin la adecuada autenticación se previene el acceso a usuarios no autorizados.

A8 - Falsificación de petición en sitios cruzados (CSRF)

WordPress usa tokens criptográficos, llamados nonces13, para validar intentos de solicitudes de acciones de usuarios autorizados para proteger contra amenazas potenciales de CSRF. WordPress proporciona una API para la generación de estos tokens para crear y verificar tokens únicos y temporales, y el token se limita a un usuario especifico, a una acción específica, a un objeto específico y a un periodo de tiempo específico, que puede añadirse a formularios y URLs según se necesite. Adicionalmente, todos los nonces son invalidados al salir del sistema.

A9 - Uso de componentes con vulnerabilidades conocidas

El equipo principal de WordPress monitoriza de cerca las pocas bibliotecas y frameworks que integra WordPress en su núcleo. En el pasado, el equipo principal ha hecho contribuciones a varios componentes de terceras partes para hacerlas más seguros, tales como la actualización para arreglar una vulnerabilidad de XSS en TinyMCE en WordPress 3.5.214.

Si es necesario, el equipo principal puede decidir realizar una versión propia o un reemplazo de componentes críticos externos, tales como la librería SWFUpload, cuando fue oficialmente reemplazada por Plupload en la 3.5.2, y una versión propia segura de SWFUpload se hizo disponible de manera inmediata por el equipo de seguridad 15 para aquellos plugins que continuaban usando SWFUpload.

A10 - Redirecciones no validadas y reenvíos

El sistema de control de acceso y autenticación interno de WordPress protegerá a los usuarios contra intentos de dirigirlos a destinos no deseados o redirecciones automáticas. Esta funcionalidad está también disponible para desarrolladores de plugins mediante una API, wp_safe_redirect()16.

Riesgos de seguridad adicionales y relaciones

Ataques de procesado XXE (XML eXternal Entity)

Cuando se procesa XML, WordPress deshabilita la carga de entidades XML personalizadas para prevenir ataques de Entidad Externa y Expansión de Entidad. Aparte de la funcionalidad del core de PHP, WordPress no proporciona ninguna API de seguridad adicional para XML para los autores de plugins.

Ataques SSRF (Server Side Request Forgery) (Solicitud falsa del lado del servidor)

La solicitudes HTTP iniciadas por WordPress son filtradas para prevenir acceso a las IPs privadas y propias. Además, el acceso se permite sólo a ciertos puertos estándar de HTTP.

Seguridad de WordPress en Plugin y Temas

El Tema por defecto

WordPress requiere de un tema que esté habilitado para presentar contenido visible en la parte pública. El tema por defecto que viene con el núcleo de WordPress (en la actualidad "Twenty Fifteen") ha sido revisado exhaustivamente y probado por razones de seguridad tanto por el equipo de desarrolladores de tema como por el equipo de desarrollo principal.

El tema por defecto puede servir como punto de partida para el desarrollo de un tema personalizado, y los desarrolladores de sitios pueden crear un tema hijo que incluya alguna personalización, pero en donde la mayor parte de la funcionalidad y la seguridad recaiga sobre el tema por defecto. El tema por defecto puede ser eliminado fácilmente por un administrador si no se necesita.

Repositorios de Temas y Plugins en WordPress.org

Hay más de 30.000 plugins y 2.000 temas listados en el sitio WordPress.org. Estos temas y plugins son enviados para su incorporación y son manualmente revisados por voluntarios antes de hacerlos disponibles en el repositorio.

La inclusión de plugins y temas en el repositorio no es una garantía de que estén libres de vulnerabilidades de seguridad. Se proporcionan guías de consulta para autores de plugin antes de su envío para inclusión en el repositorio17, y se proporciona extensa documentación sobre cómo desarrollar temas de WordPress18 en el sitio WordPress.org

Cada plugin y tema tiene la capacidad de ser desarrollada continuamente por el propietario del plugin o del tema, y cualquier arreglo posterior o desarrollo de características puede subirse al repositorio y hacerse disponible a los usuarios que tengan instalado ese plugin o tema con una descripción de ese cambio. Los administradores de los sitios son notificados de los plugins que necesitan ser actualizados mediante el panel de administración.

Cuando se descubre una vulnerabilidad en un plugin por parte del equipo de Seguridad de WordPress, contactan con el autor del plugin y trabajan conjuntamente para arreglar y liberar una versión segura del plugin. Si hay una falta de respuesta del autor del plugin, o si la vulnerabilidad es severa, el plugin/theme se retira del directorio público y, en algunos casos, arreglado y actualizado directamente por el Equipo de Seguridad.

El Equipo de revisión de Temas

El equipo de revisión de Temas es un grupo de voluntarios, liderados por miembros claves y establecidos de la Comunidad de WordPress, que revisan y aprueban los temas enviados para ser incluidos en el directorio oficial de Temas de WordPress. El Equipo de revisión de Temas mantiene la Guía oficial de Revisión de Temas19, la unidad de pruebas de datos del Tema20, y el plugin de comprobación de temas21, e intenta involucrar y educar a la comunidad de desarrolladores de Temas de WordPress en lo concerniente a las mejores prácticas en el desarrollo. La inclusión en el grupo es moderada por los commiters principales del equipo de desarrollo de WordPress.

El rol del proveedor de alojamiento en la seguridad de WordPress

WordPress puede ser instalado en multitud de plataformas. Aunque el núcleo de software de WordPress proporciona varios métodos para operar con webs seguras descritos en el presente documento, la configuración del sistema operativo y del servidor web en el que está alojado es igualmente importante para mantener la seguridad de las aplicaciones WordPress.

Una nota sobre la seguridad de WordPress.com y WordPress

WordPress.com es la instalación de WordPress más grande del mundo, y es propiedad y la gestiona la empresa Automattic, fundada por Matt Mullenweg, coautor del proyecto WordPress. WordPress.com se ejecuta sobre el núcleo de software de WordPress, y tiene sus propios procesos de seguridad, riesgos y soluciones22. El presente documento se refiere a la seguridad cuando la gestión del alojamiento es propia, descargando el software de fuentes abiertas de WordPress disponible en WordPress.org, e instalable en cualquier servidor del mundo.

Apéndice

Las APIs del núcleo de WordPress

La Interfaz de Programación de Aplicaciones (API) del núcleo de WordPress se compone de varias APIs individuales 23, donde cada una cubre unas funciones específicas y usan un conjunto dado de funcionalidades. Juntas, forman la interfaz de proyecto que permite que los plugins y los temas interactuen con, modifiquen, y extiendan, la funcionalidad del núcleo de WordPress de manera segura.

Mientras que cada API de WordPress proporciona la mejores prácticas y formas estandarizadas con las que interactuar y extender el software del núcleo de WordPress, las siguientes APIs de WordPress son las más pertinentes para reforzar y aumentar la seguridad de WordPress:

API de base de datos

La API de base de datos24, añadida en WordPress 0.71, proporciona el método correcto para acceder a los datos a través de variables que se almacenan en la capa de la base de datos.

API del sistema de archivos

La API del sistema de archivos25, añadida en WordPress 2.626, fue originalmente creada como característica propia de WordPress para sus actualizaciones automáticas. La API del sistema de archivos abstrae la funcionalidad que se necesita para leer y escribir ficheros locales del sistema de archivos para que se haga de manera segura, en una variedad de distintos tipos de alojamientos.

Lo hace a través de la clase WP_Filesystem_Base, y varias subclases que implementan distintas maneras de conectar al sistema de archivos local, dependiendo del soporte del alojamiento. Cualquier tema o plugin que necesite escribir ficheros localmente debebería hacerlo usando la familia de clases de WP_Filesystem.

API HTTP

La API HTTP27, que se añadió en WordPress 2.728 y se amplió más en WordPress 2.8, estandariza las peticiones HTTP en WordPress. La API gestiona cookies, codificación y decodificación gzip, decodificación por trozos (si es HTTP 1.1), y otras implementaciones varias de protocolos HTTP. La API estandariza las solicitudes, prueba cada método antes del envío y, basado en la configuración del servidor, usa el método apropiado para hacer la solicitud.

API de permisos y usuario actual

La API de permisos y usuario actual29 es un conjunto de funciones que ayudarán a verificar los permisos y autorización del usuario actual para realizar cualquier tarea u operación que solicite, y puede proteger además contra accesos o ejecución no autorizada por parte de usuarios no autorizados.
